# AI 사주 서비스 — AI 엔지니어링 계획서

> 팀 공유용 / v1 / 작성: AI 엔지니어링 파트
> 목적: 모델 선정·프롬프트 설계·궁합 데이터 구조에 대한 의사결정을 팀과 합의하기 위함

---

## 0. 한 장 요약

- **이 서비스의 AI 난이도는 "LLM이 사주를 잘 푸는 것"이 아니라, 사주 계산(만세력)을 LLM 바깥에서 결정론적으로 처리하고 LLM에게는 해석만 시키는 분리 구조를 만드는 데 있다.**
- LLM에게 사주팔자 자체를 계산시키면 거의 다 틀린다. 신뢰도(기획서 6장 "신뢰감 중요")가 여기서 무너진다.
- 모델은 하나로 통일하지 않는다. **용도별 2티어**로 간다.
  - 유료 리포트(매출 핵심): 고품질 모델
  - 오늘의 운세 / 무료 챗봇(호출 많음): 저비용 모델
- 무료 기능을 매번 LLM으로 처리하면 비용이 사용자 수에 비례해 터진다. **캐싱이 곧 비용 전략이다.**

## 1. 핵심 아키텍처 결정: 계산과 해석의 분리

### 1-1. 왜 LLM에게 사주를 계산시키면 안 되는가

LLM에 "1995년 3월 5일 오전 9시 서울 출생 사주를 뽑아줘"라고 하면 천간지지를 그럴듯하게 출력한다. 그러나 이는 토큰 확률로 "흉내"낸 것이며 실제 계산이 아니다. 만세력 계산은 다음을 요구한다.

- 음양력 변환
- 절기(節氣) 기준 월주 결정
- 진태양시(경도 기반) 보정
- 서머타임 이력 반영

특히 절입(節入) 시각 근처 출생자는 월주가 통째로 바뀌는데, LLM은 이를 거의 맞히지 못한다. 사주를 아는 사용자는 여덟 글자가 틀린 것을 즉시 알아챈다. 서비스 신뢰도의 핵심 리스크다.

### 1-2. 채택 구조

```
[생년월일시 입력]
      │
      ▼
[만세력 엔진]  ← Python, 결정론적 계산. LLM 아님.
      │  (사주팔자 + 오행/십성/대운 구조화 JSON)
      ▼
[LLM]  ← 위 JSON을 "해석"만 함. 계산 금지.
      │
      ▼
[리포트 / 오늘의 운세 / 챗봇 답변]
```

- LLM의 역할: "갑목 일간이 신금 정관을 만났다 → 어떤 의미인가"를 자연어로 풀이.
- LLM이 절대 하지 않는 것: 사주팔자 글자 계산.

### 1-3. 만세력 엔진은 직접 구현하지 않고 검증된 라이브러리를 쓴다

- `sxtwl` (寿星天文历): 절기 기준 간지 계산. 중화권 사주 서비스 다수가 사용.
- `korean_lunar_calendar`: 한국천문연구원 데이터 기반 음↔양 변환.
- 진태양시 보정(경도 기반 ±, 표준자오선 135°E 기준)만 자체 로직으로 추가.

이 엔진 출력의 정확성이 그 위에 올린 모든 LLM 해석의 신뢰 기반이다. 이 부분만큼은 출시 전 수기 검증(실제 만세력 사이트 대조)을 반드시 거친다.

## 2. 사용 추천 LLM 및 가격 비교

> 가격은 2026년 6월 기준, 100만 토큰(MTok)당 USD. 입력/출력 분리 과금.
> LLM 가격은 자주 바뀌므로 발주·계약 전 공식 페이지 재확인 필수.

### 100만 토큰(1 MTok)은 어느 정도의 출력 분량인가요?

- 글자 수 변환: LLM의 한국어 토크나이저 효율에 따라 다르지만, 보통 100만 토큰은 약 50만~70만 글자(띄어쓰기 포함) 분량의 한국어 텍스트를 생성할 수 있습니다.
- 서비스 체감 분량: 유료 심화 사주 리포트 1건의 출력 길이를 약 1,500자(약 2,000~2,500토큰)로 기획한다고 가정할 때, 출력 100만 토큰당 약 400~500건의 유료 리포트를 생성할 수 있습니다.

### 2-2. 비용 절감 공통 레버

- 배치 API: 비실시간 작업 50% 할인 (오늘의 운세 사전 생성에 적합).
- 프롬프트 캐싱: 반복되는 system 프롬프트·공통 지시문 캐싱으로 입력 비용 절감.

### 2-3. 모델 운용 가설

- 오늘의 운세 / 무료 챗봇 → Gemini Flash 계열 (단가 최저, 대량 호출 대응)
- 유료 심화 리포트 → Claude, Gemini Pro, GPT 중 블라인드 비교 승자 채택.

## 3. 프롬프트 엔지니어링 설계

### 3-1. 대원칙

> "계산시키지 말고, 계산된 결과를 해석시켜라."

모든 프롬프트는 사주 데이터(JSON)를 **입력으로 주입**받아 해석만 수행한다. 데이터에 없는 천간/지지/십성을 임의 생성하지 않도록 명시 가드를 건다.

### 3-2. LangChain 기반 프롬프트 파이프라인 모듈화

- 7종 리포트(연락운, 궁합, 재회운 등)의 공통 골격과 주제별 관점 블록을 LangChain을 활용해 모듈화하여 관리한다.
- 안정적인 JSON 출력: content_free와 content_locked 구분을 Pydantic 객체로 정의하여 Output Parser에 연동한다. 포맷 이탈이나 환각 발생 시 자동 재시도(Retry) 로직이 작동하도록 구성한다.

### 3-3. 리포트 프롬프트 골격

```
[System]
당신은 정통 명리학 기반 사주 해석가입니다.
당신은 사주를 계산하지 않습니다. 제공되는 사주 데이터는 이미 정확히
계산된 값이며, 당신의 역할은 이를 해석하는 것입니다.
- 데이터에 없는 천간/지지/십성을 만들지 마세요.
- 단정적 예언(반드시 ~한다)을 피하고 경향성으로 서술하세요.
- 의료/법률/투자/생사에 대한 단정 조언을 하지 마세요.

[User]
## 사주 데이터 (계산 완료)
- 사주팔자: 을해년 무인월 갑자일 병인시
- 일간: 갑목 / 오행 분포: 목3 화2 토1 금1 수1
- 십성: 정관2 식신1 편재1
- 현재 대운: 32~41세 신유 대운
- 분석 주제: {연애 패턴}
- 사용자 입력 고민: {선택}

## 출력 형식 (반드시 준수)
[무료 영역]
1. 현재 흐름 요약 (2~3문장)
2. 핵심 결론 한 줄
[유료 영역]
3. 유형/점수
4. 상세 분석
5. 주의할 행동 / 추천 행동 가이드
6. 기간 흐름(7일/30일)
```

### 3-4. 설계 포인트

- **무료/유료 섹션 분리 출력**: 프론트 블러 처리를 위해 JSON 키로 `content_free` / `content_locked`를 나눠 받는다. "결론은 무료, 상세는 유료" 정책(기획서 6-5)을 출력 구조로 강제.
- **주제별 템플릿화**: 연락운·궁합·재회운은 출력 구조가 같고 관점만 다르다. 공통 골격 + 주제별 관점 블록으로 관리해 7종 리포트를 코드 중복 없이 생성.
- **온도(temperature)**: 리포트 ~0.7(맞춤형 느낌), 오늘의 운세 낮게(일관성).
- **안전 가드**: 흉운/건강·생사 단정 금지를 system에 명시. 사주 서비스의 평판 리스크 1순위.
- **궁합/최애 케미**: 두 사람 사주 JSON을 함께 주입. 시간 모름(시주 없음) 분기를 프롬프트에 명시.

## 4. 모델 선정 방법: 블라인드 출력 비교

가격표만으로 유료 리포트 모델을 정할 수 없다. 다음 절차로 팀이 함께 고른다.

1. 동일한 사주 데이터 + 동일 프롬프트를 후보 모델 3종(예: Claude Sonnet / Gemini Pro / GPT)에 동시 투입.
2. 모델명을 가린 상태로 출력 3개를 나란히 본다.
3. 팀이 다음 기준으로 평가:
   - 사주 용어의 자연스러움
   - 한국어 서술의 결 / 타깃(20~30대 여성) 톤 적합성
   - 출력 포맷 준수도 (무료/유료 분리, 길이)
   - 환각(데이터에 없는 글자 생성) 여부
4. 출력 품질이 비슷하면 단가가 낮은 쪽을 택한다.
5. 최종 후보 2개는 실제 사용자 반응(완독률, 결제 전환)으로 A/B 한다.

> 이 비교를 바로 해볼 수 있는 인터랙티브 도구를 함께 제공한다 (별도 파일).

## 5. 궁합 / 최애 케미 — 데이터·API 관점

### 5-1. 궁합 (사용자 ↔ 상대)
- 상대 생년월일시를 입력받아 두 사주를 각각 엔진에 넣고, 두 JSON을 LLM에 함께 주입.
- 외부 API 불필요. 시간 모름 케이스(기획서 사전 정보 입력) 분기 필수.

### 5-2. 최애 케미 (사용자 ↔ 연예인)
- 연예인은 보통 생년월일 공개 / 출생시간 비공개 → **시주 제외 약식 궁합**으로 설계.
- 외부 운세 API에 의존하지 않고, **직접 작은 연예인 테이블을 관리**한다.
  - `celebrities(id, name, group, birth_date, has_birth_time, gender)`
  - 인기 인물 수백 명의 생년월일을 위키 등에서 모으는 일회성 데이터 작업.
- **법적 가드**: 실존 인물에 대한 단정적·부정적 서술은 명예훼손 리스크. 기획서대로 "가상 케미" 엔터테인먼트 프레이밍을 프롬프트에 명시해, 인물 평가가 아니라 사주 궁합 놀이로 출력되게 한다.

## 6. 비용 폭탄 방지: 캐싱 전략 (가장 중요한 운영 결정)

무료 기능(오늘의 운세, 1일 3개 무료 질문)을 매 호출마다 LLM으로 처리하면, **매출 없는 토큰 비용이 사용자 수에 비례해 증가**한다. 이 서비스에서 과금이 터질 1순위 지점이다.

| 콘텐츠 | 캐싱 단위 | 효과 |
|---|---|---|
| 오늘의 운세 | (날짜 + 일간 + 오늘일진) 조합 | 사용자 10만명이어도 LLM 호출은 조합 수(수십 회)로 끝 |
| 정통사주 풀이 | 사용자 사주(불변) | 1회 생성 후 영구 저장, 재방문 시 재호출 0 |
| 심화 리포트 | 사용자×주제×상대 | 생성 후 저장, 재열람·이메일 발송 재사용 |

- 오늘의 운세는 **배치 API로 새벽에 사전 생성** + 캐싱하면 단가도 절반, 호출도 최소.
- 정통사주·리포트는 결과를 DB에 저장(이미 백엔드 스키마에 반영됨).

## 7. 팀 의사결정 요청 사항

1. 유료 리포트용 모델: 블라인드 비교(4장) 일정 잡기. 후보 = Claude Sonnet 4.6 / Gemini Pro / GPT 중.
2. 저비용 티어 모델: Gemini Flash 계열로 잠정 채택 동의 여부.
3. 연예인 데이터 수집: 1차 대상 인물 리스트 누가 정리할지.
4. 안전 가드 문구(흉운·생사·실존인물): 최종 톤을 기획/법무와 확정.
5. 캐싱 정책: 오늘의 운세 배치 사전생성 시간대(예: 매일 0시) 합의.
